Creating custom R Shiny inputs and outputs: A comprehensive guide

R Shiny’s binding system enables developers to create fully custom interactive components by bridging JavaScript and R through a WebSocket-based communication protocol. Custom input bindings allow form widgets to send user data to R, while output bindings render server-side computations in the browser. This bidirectional architecture—powered by the Shiny.InputBinding and Shiny.OutputBinding classes—provides complete control over component behavior, from simple toggles to complex D3.js visualizations. The framework has matured significantly, with authoritative documentation from Posit and several expert-authored books providing detailed implementation guidance.

How Shiny’s input/output system works under the hood

Shiny uses WebSockets for real-time, bidirectional communication between the R server and the JavaScript client, powered by the httpuv package. When a Shiny app loads, an HTTP request initiates a WebSocket handshake (HTTP status code 101 switches the protocol from HTTP to WS/WSS), establishing a persistent connection for continuous two-way messaging.

The communication flow follows a consistent pattern: input changes travel from browser to R server, while output updates flow from R server to browser. Key functions driving this system include session$sendCustomMessage() for R-to-JavaScript messaging, session$sendInputMessage() for updating input values, and Shiny.setInputValue() on the JavaScript side to set input values programmatically.

All input and output components are managed through two central registries: Shiny.inputBindings and Shiny.outputBindings. These registries use a priority system where higher numbers indicate higher priority (default is 0), allowing developers to override built-in bindings when necessary. When Shiny initializes a page, it calls Shiny.bindAll() to scan the DOM and associate each matching element with its appropriate binding.

Integrating JavaScript, CSS, and Shiny applications

The www/ directory and static assets

The www/ directory at your Shiny app’s root serves as the standard location for static files. Files placed here are automatically accessible at URL paths without the www/ prefix—for example, www/js/myBinding.js becomes available at /js/myBinding.js. In R packages, use inst/www/ and access files via system.file('www', package='pkg').

ui <- fluidPage(
  tags$head(
    tags$script(src = "js/myBinding.js"),
    tags$link(rel = "stylesheet", href = "css/myStyles.css")
  )
)

Using htmlDependency for robust asset management

For packages and reusable components, htmltools::htmlDependency() provides the recommended approach. This function automatically handles dependency deduplication—if the same dependency appears multiple times on a page, Shiny includes it only once, selecting the highest version when conflicts arise.

myDependency <- function() {
  htmltools::htmlDependency(
    name = "myComponent",
    version = "1.0.0",
    src = c(file = system.file("www", package = "myPackage")),
    script = "js/myBinding.js",
    stylesheet = "css/myStyles.css"
  )
}

myInput <- function(inputId, label, value = "") {
  tagList(
    myDependency(),
    tags$div(
      id = inputId, 
      class = "my-input-class",
      tags$label(label),
      tags$input(type = "text", value = value)
    )
  )
}

The singleton() wrapper offers an alternative for simpler cases, ensuring that wrapped content appears only once regardless of how many times the function is called. For package development, addResourcePath() creates named URL prefixes mapping to local directories.

The Shiny.InputBinding class and its methods

Custom input bindings extend Shiny.InputBinding using jQuery’s $.extend() pattern. All instances of the same input type share a single binding object, with individual elements passed as parameters to each method.

Required methods for input bindings

The find(scope) method locates all instances of your input component within a given scope (usually the document). It must return an array-like object of matching elements, typically using jQuery’s class-based selectors to support multiple instances:

find: function(scope) {
  return $(scope).find('.my-input-class');
}

The getValue(el) method extracts and returns the current value to send to R. This function must include an explicit return statement—omitting it causes the binding to fail silently. Values can be primitives or objects (which become lists in R):

getValue: function(el) {
  return {
    value: $(el).val(),
    collapsed: $(el).hasClass('collapsed-box')
  };
}

The subscribe(el, callback) method sets up event listeners to detect value changes. Use jQuery event namespacing (e.g., change.myBinding) for clean unsubscription. The callback function accepts an optional boolean parameter—passing true applies the rate policy defined in getRatePolicy():

subscribe: function(el, callback) {
  $(el).on("change.myBinding", function(e) {
    callback();  // Immediate send
  });
  $(el).on("keyup.myBinding", function(e) {
    callback(true);  // Apply rate policy
  });
}

Optional methods for enhanced functionality

The setValue(el, value) method programmatically sets the input’s value, essential for updateInput functions. The receiveMessage(el, data) method handles messages sent from R via session$sendInputMessage(), typically calling setValue() and triggering a change event:

receiveMessage: function(el, data) {
  if (data.hasOwnProperty('value')) {
    this.setValue(el, data.value);
    $(el).trigger('change');
  }
}

The getRatePolicy() method controls update frequency with three policies: "direct" sends immediately, "debounce" waits until no new values arrive for the specified delay (ideal for text inputs), and "throttle" limits updates to at most one per delay period (ideal for sliders):

getRatePolicy: function() {
  return { policy: 'debounce', delay: 250 };
}

The getType() method returns a string identifying a custom input handler for R-side data deserialization, connecting to handlers registered via shiny::registerInputHandler(). The initialize(el) method runs once per element before other methods, essential for initializing third-party JavaScript libraries.

Complete input binding registration

var myBinding = new Shiny.InputBinding();
$.extend(myBinding, {
  find: function(scope) { return $(scope).find('.my-input'); },
  getValue: function(el) { return $(el).val(); },
  setValue: function(el, value) { $(el).val(value); },
  subscribe: function(el, callback) {
    $(el).on('change.myBinding keyup.myBinding', function() { callback(true); });
  },
  unsubscribe: function(el) { $(el).off('.myBinding'); },
  receiveMessage: function(el, data) {
    if (data.value !== undefined) {
      this.setValue(el, data.value);
      $(el).trigger('change');
    }
  },
  getRatePolicy: function() { return { policy: 'debounce', delay: 250 }; }
});

Shiny.inputBindings.register(myBinding, 'mypackage.myInput');

Building complex form widgets with custom bindings

Multi-select components

The shinyWidgets package demonstrates effective patterns for multi-select inputs. The key is returning an array of selected values (or null for empty selections) and handling the DOM events from the underlying library:

var multiInputBinding = new Shiny.InputBinding();
$.extend(multiInputBinding, {
  find: function(scope) { return $(scope).find(".multi-input"); },
  getValue: function(el) {
    var selected = [];
    $(el).find("option:selected").each(function() {
      selected.push($(this).val());
    });
    return selected.length === 0 ? null : selected;
  },
  subscribe: function(el, callback) {
    $(el).on("change.multiInput", function() { callback(); });
  }
});

Composite inputs tracking multiple properties

For complex widgets like collapsible boxes that need to track multiple state properties, embed configuration in a <script type="application/json"> tag and return an object from getValue():

getValue: function(el) {
  var config = JSON.parse($(el).find('script[data-for="' + el.id + '"]').html());
  return {
    collapsed: $(el).hasClass('collapsed-box'),
    width: config.width,
    title: config.title
  };
}

Input validation with shinyvalidate

Custom bindings can implement two methods for validation display. The setInvalid(el, data) method displays validation errors, receiving an object with type and message properties. The clearInvalid(el) method removes error displays:

setInvalid: function(el, data) {
  $(el).addClass('is-invalid');
  $(el).after('<span class="validation-message text-danger">' + data.message + '</span>');
},
clearInvalid: function(el) {
  $(el).removeClass('is-invalid');
  $(el).parent().find('.validation-message').remove();
}

The Shiny.OutputBinding class for custom outputs

Output bindings follow a simpler pattern than inputs, primarily requiring find() and renderValue() methods.

Core output binding methods

The find(scope) method works identically to input bindings, locating output elements by class. The renderValue(el, data) method receives data from R (converted to JSON via shiny:::toJSON()) and renders it to the element:

var myOutputBinding = new Shiny.OutputBinding();
$.extend(myOutputBinding, {
  find: function(scope) { return $(scope).find('.my-output'); },
  renderValue: function(el, data) {
    el.style.backgroundColor = data.color;
    document.getElementById(el.id + '-title').innerText = data.title;
  },
  renderError: function(el, error) {
    el.innerHTML = '<span class="error">' + error.message + '</span>';
  },
  clearError: function(el) {
    $(el).find('.error').remove();
  }
});
Shiny.outputBindings.register(myOutputBinding, 'mypackage.myOutput');

R-side components for custom outputs

Creating a custom output requires three R functions: a UI generator, a render function using exprToFunction(), and optionally an HTML dependency function:

myOutput <- function(outputId, width = '100%', height = '400px') {
  tagList(
    myOutputDependency(),
    tags$div(id = outputId, class = 'my-output', 
             style = sprintf('width: %s; height: %s;', width, height))
  )
}

renderMyOutput <- function(expr, env = parent.frame(), quoted = FALSE) {
  func <- exprToFunction(expr, env, quoted)
  function() {
    val <- func()
    list(title = val$title, color = val$color)
  }
}

Integrating D3.js and interactive visualizations

Using the r2d3 package

The r2d3 package provides the most streamlined approach to D3.js integration, handling the binding infrastructure automatically:

library(r2d3)
output$d3 <- renderD3({
  r2d3(runif(5, 0, input$max_value), script = "barchart.js")
})

Within D3 scripts, communicate back to Shiny using Shiny.setInputValue():

svg.selectAll("rect").data(data)
  .enter().append("rect")
  .on("click", function(d) {
    Shiny.setInputValue("bar_clicked", d, {priority: "event"});
  });

The htmlwidgets framework

The htmlwidgets package provides a comprehensive framework for creating R bindings to JavaScript libraries that work seamlessly across R console, R Markdown, Shiny, and standalone HTML. Widgets use a factory function pattern returning renderValue() and resize() methods:

HTMLWidgets.widget({
  name: 'mywidget',
  type: 'output',
  factory: function(el, width, height) {
    var instance = new MyLibrary(el.id);
    return {
      renderValue: function(x) {
        instance.setData(x.data);
        instance.render();
      },
      resize: function(width, height) {
        instance.resize(width, height);
      }
    };
  }
});

Major packages built on htmlwidgets include plotly for interactive charts, leaflet for maps, DT for tables, dygraphs for time series, and networkD3 for network visualizations.

Handling responsive resizing

The sizing policy in htmlwidgets controls how widgets respond to container changes across different contexts. For custom output bindings, listen for window resize events and update visualizations accordingly:

resize: function(width, height) {
  d3.select(el).select("svg")
    .attr("width", width)
    .attr("height", height);
  force.size([width, height]).resume();
}

Advanced communication patterns between R and JavaScript

Shiny.setInputValue() with priority options

By default, Shiny optimizes by ignoring duplicate values. To treat every call as an event regardless of value:

Shiny.setInputValue("myevent", value, {priority: "event"});

Custom input handlers enable type-specific deserialization. Register handlers in your package’s .onLoad() function and reference them with a colon suffix in JavaScript:

registerInputHandler("mypackage.dateTime", function(x, session, inputname) {
  as.POSIXct(x, origin = "1970-01-01")
})
Shiny.setInputValue("timestamp:mypackage.dateTime", Date.now());

Custom message handlers for R-to-JavaScript communication

The session$sendCustomMessage() function sends arbitrary data from R to JavaScript, paired with handlers registered via Shiny.addCustomMessageHandler():

# R server
observe({
  session$sendCustomMessage("update-chart", list(data = processed_data))
})
// JavaScript
Shiny.addCustomMessageHandler("update-chart", function(message) {
  updateVisualization(message.data);
});

Important for modules: sendCustomMessage() does not automatically namespace message types. When using modules, manually include namespaced IDs in the message payload:

# In module server
session$sendCustomMessage("handler", list(id = ns("target_input"), value = data))

Proper DOM updates with rebinding

When dynamically modifying the DOM, ensure proper binding lifecycle management:

Shiny.unbindAll(scope);
// ... modify DOM ...
Shiny.initializeInputs();
Shiny.bindAll(scope);

For content insertion, use Shiny.renderContent() which handles this automatically.

Testing and debugging custom bindings

Integration testing with shinytest2

The shinytest2 package supports testing custom bindings. For inputs without standard bindings, use the allow_no_input_binding_ parameter:

test_that("Custom input works", {
  app <- AppDriver$new()
  app$set_inputs(table_rows_selected = 1, allow_no_input_binding_ = TRUE)
  app$expect_values()
})

For robust tests, add data-testid attributes to components and locate elements by this stable identifier rather than potentially changing IDs.

Debugging techniques

Enable comprehensive R-side debugging with options(shiny.trace = TRUE) to view WebSocket traffic, showing all messages between browser and server. The reactive log (reactlog::reactlog_enable()) visualizes dependency relationships.

For JavaScript debugging, use browser developer tools with the debugger; statement to pause execution. In RStudio Desktop on macOS, enable WebKit Developer Tools with:

defaults write org.rstudio.RStudio WebKitDeveloperExtras -bool true

Performance optimization

Implement getRatePolicy() to prevent flooding the server with rapid updates. For bindings wrapping third-party libraries that trigger many events, use manual debouncing:

subscribe: function(el, callback) {
  var timeout = null;
  $(el).on('slide', function() {
    if (timeout) clearTimeout(timeout);
    timeout = setTimeout(function() { callback(); }, 100);
  });
}

Clean up properly in unsubscribe() to prevent memory leaks:

unsubscribe: function(el) {
  $(el).off('.myBinding');
  $(el).removeData('myPlugin');
}

Key packages demonstrating best practices

The shinyWidgets package by Victor Perrier and Fanny Meyer provides dozens of production-quality input bindings including multi-selects, sliders, and switches. Its source code demonstrates comprehensive implementation patterns for complex widgets. The shinyjs package by Dean Attali shows how to execute arbitrary JavaScript from R without complex binding infrastructure.

The htmlwidgets package, maintained by Carson Sievert at Posit, provides the foundational framework that powers plotly, leaflet, and DT. For React integration, reactR by Kent Russell creates htmlwidgets using React components, while Appsilon’s shiny.react offers an alternative approach for building React-based Shiny components.

Conclusion

Creating custom Shiny bindings requires understanding three interconnected systems: the R-side functions generating HTML and handling messages, the JavaScript binding classes managing DOM interaction, and the WebSocket protocol connecting them. The Shiny.InputBinding class with its find(), getValue(), subscribe(), and receiveMessage() methods forms the foundation for all custom inputs, while Shiny.OutputBinding with renderValue() handles custom outputs.

For production applications, prioritize using htmltools::htmlDependency() for asset management, implement proper rate limiting, handle validation states, and ensure cleanup in unsubscribe(). The htmlwidgets framework significantly reduces boilerplate for visualization outputs. Testing with shinytest2 and debugging with options(shiny.trace = TRUE) catch issues before deployment. As the ecosystem matures, the posit-dev/shiny-bindings repository provides emerging patterns for React component integration, pointing toward increasingly sophisticated component architectures.
